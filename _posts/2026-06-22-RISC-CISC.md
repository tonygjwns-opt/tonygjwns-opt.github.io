고급 컴퓨터 구조 수업 과제로 읽은 두 논문 — "The Case for the Reduced Instruction Set Computer"(RISC 옹호)와 그에 대한 반박(CISC/VAX 측) — 을 쟁점별로 정리했다. 현대엔 RISC가 더 효율적이라는 게 정설이지만, 당시엔 어느 쪽도 자명하지 않았다는 점이 흥미롭다.

## CISC는 왜 복잡해졌나 (CISC 측 논리)

- **메모리–CPU 속도차**: CPU가 메모리보다 빨라, 복잡한 고수준 명령 하나로 메모리 접근을 줄이는 게 이득이라 봤다.
- **마이크로코드**: microcode 덕에 명령을 추가하는 비용이 작다(control store 몇 word) → 서브루틴을 아키텍처에 넣는 방향.
- **코드 밀도**: 메모리가 비싸 compact한 프로그램을 선호.
- **상위 호환**: 새 명령은 추가만 하고 기존 건 안 지움 → 계속 비대해짐.
- 그 밖에 HLL(고급 언어) 지원, 마케팅("더 많은 연산"), 멀티프로그래밍(인터럽트 시 저장할 state 증가) 등.

## 핵심 반박 (RISC 측)

- **속도차 전제 붕괴**: 반도체 메모리로 CPU–메모리 속도차가 줄어 복잡 명령의 명분이 약해졌다.
- **명령 추가 비용 논리 오류**: 같은 instruction set이어도 microcode 길이는 달라 비교가 부정확. 개발은 1회지만 실행은 다회 → 실행 효율이 우선.
- **복잡 ≠ HLL 지원**: 복잡할수록 오히려 컴파일러 작성자 부담이 커진다. 한 연산을 실행하는 방법이 여러 개로 늘고, 'because it is there' 식으로 안 쓰이는 명령이 생긴다.
- **사용 통계**: 실제로 소수 명령이 대부분을 차지. 단 '횟수'가 아니라 '사용 시간' 기준으로 보면 순위가 바뀌고, 그게 더 중요하다.
- **특수 명령이 항상 최선은 아님**: 간단한 명령 몇 개 조합이 더 빠른 경우도 있다(load, INDEX 등).

## RISC의 이점

- **구현 용이**: 단순해서 전체 CPU를 한 칩에 넣기 쉽고, 남는 칩 공간을 파이프라이닝 등에 써 더 빨라진다.
- **디자인 시간 단축**: 칩 발전 속도에 비해 복잡한 설계는 시간이 너무 걸린다.
- (반박도 있음: "단순 = 빠름"의 직접 근거는 약하다는 지적, 그리고 애초에 'Reduced'를 어떻게 정의하느냐가 모호하다는 지적.)

## 남는 교훈

- 현대엔 RISC가 정설(ARM 등)이지만, 당시엔 양쪽 다 합리적 근거가 있었다. "지금의 정답이 그때도 자명했던 건 아니다."
- 논쟁의 축: 복잡한 명령으로 메모리 접근·코드 밀도를 아낄 것인가(CISC) vs 단순한 명령으로 구현·속도·칩 활용을 얻을 것인가(RISC). 전제(속도차·메모리 비용)가 바뀌면 답도 바뀐다.
