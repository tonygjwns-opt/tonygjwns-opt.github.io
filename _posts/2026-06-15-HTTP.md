## 본질

- **HTTP**(HyperText Transfer Protocol): 웹 브라우저(client)와 웹 서버가 인터넷에서 소통할 때 쓰는 요청·응답 규약.
- 실제 데이터 이동은 **TCP/IP**가 담당. HTTP는 application 레벨의 추상 규약(무엇을 어떻게 주고받을지의 표준)일 뿐, 자체로 데이터를 나르지는 않음 → 여전히 TCP/IP 위에서 request·response가 오감.

## 1. 핵심 용어

- **Client** (크롬·Firefox 등 브라우저): 내 컴퓨터에서 돌며 인터넷에 연결됨. 사용자의 상호작용을 서버로 보낼 request로 번역. 내 컴퓨터 통째로를 client로 봐도 무방. 식별용 고유 주소(IP) 보유.
- **Server**: 인터넷 연결 + IP 보유. request를 기다렸다 응답. client와 다른 점은 들어온 request에 어떻게 반응할지 아는 '서버 소프트웨어'가 설치된 것. 주 기능 = client에게 웹 페이지를 저장·처리·전달. (웹/파일/application/DB 서버 등.)
- **IP 주소**(Internet Protocol address): TCP/IP 네트워크에서 기기의 숫자 식별자. `.`으로 구분된 4개 숫자 set(논리 주소) → TCP/IP가 하드웨어 물리 주소로 변환.
- **ISP**(Internet Service Provider): client·server 사이 중개자(보통 케이블 회사). 주소를 받으면 DNS lookup으로 IP를 찾아옴.
- **DNS**(Domain Name System): 도메인 이름 ↔ IP를 추적하는 분산 데이터베이스.
- **Domain Name**: 하나 이상의 IP를 식별하는 이름. DNS가 받아 해당 IP를 찾음.
- **TCP/IP**(Transmission Control Protocol / Internet Protocol): 프로토콜 = 표준 규칙. 가장 널리 쓰임.
- **Port(포트)**: IP와 연관된 16비트 정수. 서버 안 특정 프로세스(채널)를 식별 → request가 어느 프로세스로 갈지 가림.
- **Host**: 네트워크에 연결된 컴퓨터(client·server 등). Server는 Host의 한 종류(특정 machine). Host는 여러 서비스를 운영하는 hosting 업체일 수도.
- **URL**(Uniform Resource Locator): 웹 자원 식별. 예 `https://github.com/someone` → 프로토콜(https) + host name(github.com) + 경로/파일.
- **HTML**(HyperText Markup Language): 태그로 문서 구조 기술. 예 `<h1>` heading, `<p>` paragraph.

## 2. 웹은 어떻게 동작하나 (URL 입력 → 페이지 로딩)

1. URL 입력(예 `https://www.github.com/`).
2. 브라우저가 URL 파싱: 프로토콜 / 도메인 / 자원경로(`/`). 경로가 비면 메인 페이지.
3. 브라우저가 ISP와 통신 → DNS lookup으로 호스팅 웹서버 IP 탐색.
4. DNS 탐색은 계층(hierarchy)을 타고 감:
   - root name server에 물어 `.com` 도메인용 name server IP를 얻음.
   - `.com` name server에 `www.github.com` 주소를 요청.
   - 최종 서버 IP를 얻으면 브라우저로 전달(못 얻으면 계층을 더 내려가며 탐색).
5. 브라우저가 IP + 포트(HTTP 기본 80, HTTPS 기본 443)로 **TCP 소켓 연결** 개시 → 브라우저-서버 연결 완성.
6. 브라우저가 메인 HTML을 요청하는 HTTP request(구체적으론 `GET`)를 보냄.
7. 서버가 HTML을 찾아 있으면 response를 보냄(없으면 404).

   ```http
   GET /index.html HTTP/1.1
   Host: www.github.com

   HTTP/1.1 200 OK
   Content-Type: text/html
   ```

8. 브라우저가 HTML을 위에서 아래로 파싱하며 Asset(CSS·이미지·JS 등) 확인.
9. 각 Asset마다 필요하면 추가 HTTP request로 받아옴(8 반복).
10. 다 받으면 페이지가 렌더링되고 연결 종료.

> HTTP/1.1은 기본적으로 연결을 재사용(**keep-alive**)해 여러 요청을 한 연결로 처리한다. 매 요청마다 새로 여는 건 HTTP/1.0.

## 3. 정보의 전달 (packet & TCP/IP)

- request 정보는 **packet**이라는 작은 덩어리로 쪼개짐.
- 각 packet에 **TCP 헤더**(출발·도착 포트)와 **IP 헤더**(출발·도착 IP)가 붙음.
- 이더넷/와이파이/셀룰러로 전달 → 목적지에서 재조립.
- **TCP** = packet으로 쪼개 정확히 보내고, 손실분 재전송, 끝에서 올바른 순서로 재조립.
- **IP** = IP 헤더로 packet을 다른 컴퓨터로 라우팅(주소 기반 전달).

## 4. 받은 자원 → 화면 렌더링

- 브라우저의 rendering engine이 HTML 파싱 알고리즘으로 파싱.
- 결과로 **DOM**(Document Object Model) 트리 생성 — node를 어떻게 배치할지 표현. JavaScript로 조작.
- DOM 생성 후 **CSS** 파싱 → DOM 노드를 훑으며 스타일·위치·좌표 계산.

## 5. client-server 모델

웹사이트·웹앱의 공통 구조 = **client + server + database**.

- **Client**(사용자가 보고 상호작용): 구조=HTML, look&feel=CSS(Cascading Style Sheet), 상호작용=JavaScript(서버 없이 처리하기도, 서버에 request 보내기도).
- **Server**: 특정 포트에서 request 수신 대기 → action 수행. 포트 = 한 컴퓨터에서 다른 task를 처리하는 분리된 채널.
- **Database**: 정보 저장·관리·갱신. 페이지에 들어가는 데이터의 출처.

## 6. 규모 확장 (scale)

- 핵심 문제는 동시다발 요청 → 백엔드 서버 그룹에 트래픽 분산.
- 분배는 DNS가 아니라 **load balancer**가 함(가장 빠르고 효율적인 서버로).
  - client에 공개하는 '가상 IP'가 load balancer를 가리킴.
  - 알고리즘: **round robin**(골고루) / **least connections**(활성 연결 최소인 서버로).
  - 커지면 session persistence 같은 문제 발생.
- **Service**: 서버들과만 상호작용하는 self-contained 기능 단위(사용자 승인·검색 등). 웹 서버를 service로 분할 → 팀별 작업 용이.
- **CDN**(Content Delivery Network): 여러 데이터센터에 분산된 프록시(중개) 서버. DNS가 본 서버 대신 CDN의 **edge server** IP로 안내, 자주 쓰는 콘텐츠는 캐싱해 본 서버 없이 전달.

## 7. HTTP 상세 & REST

**기본**

- request-response 패턴의 공통 언어·규칙이 HTTP. (syntax=양식·인코딩, semantics=규칙, timing=속도·순서.)
- 요청·응답 한 쌍 = 하나의 **HTTP transaction**. HTTP는 text 기반. 각 메시지 = **헤더 + body**.

**HTTPS = HTTP over TLS/SSL**

- HTTP는 보안에 취약 → 암호화한 것이 HTTPS. 포트 **443**. 연결 시 **TLS handshake**로 세션 key 합의.

**헤더(메타데이터)** — `F12 → Network`로 확인

- request 종류(GET/POST/PUT/DELETE), 경로(URL·remote 주소), status code, content 종류, user-agent, cookie, (가끔) post body, accept-encoding/language 등.
- 매번 직접 안 씀: 사이트 방문 시 사이트가 cookie를 세팅(보낼 헤더 포함). 쿠키 없으면 쿠키 데이터 안 보냄.

**body** — 메시지 본문(request는 빌 수도). 응답엔 핵심 데이터가 담김. `F12 → Response`에서 확인.

**메서드**

- `GET` : URL 정보를 읽음. **safe** + **idempotent**(반복해도 결과 동일, 서버 상태 불변).
- `POST` : 새 리소스 생성(가입 폼 등). 부모 리소스(URL) 아래 종속 리소스를 만듦. **비safe·비idempotent**(반복하면 매번 새로 생성).
- `PUT` : body로 URL 리소스를 갱신(없으면 생성). 상태를 바꿔 비safe, 단 **idempotent**(여러 번 해도 첫 결과와 같음).
- `DELETE` : URL 리소스 삭제. **idempotent**(반복해도 결과 동일 — 이미 삭제됐으면 404).

**REST**(Representational State Transfer)

애플리케이션 설계 가이드라인. **stateless·client-server·cacheable** 프로토콜(대개 HTTP)을 쓰라는 것. 핵심 2:

1. **일관된 인터페이스**: 리소스는 URL로 식별. client는 리소스가 아니라 **representation**과만 상호작용. 모든 메시지는 자기 설명적. state는 client(URL·query·body·헤더)↔server(body·response code·헤더)로 주고받음.
2. **무상태(stateless)**: 요청 처리에 필요한 state는 request 자체에, 응답에 필요한 state는 response 자체에 다 담음 → 서버가 요청 사이에 세션 state를 유지할 필요가 없음.

## 8. 상태코드 (참고)

앞자리: **2xx** 성공(200 OK) / **3xx** 리다이렉트 / **4xx** 요청 잘못(404) / **5xx** 서버 오류.
